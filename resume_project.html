项目背景
企业内部存在大量非结构化文档（PDF/Word/网页），传统关键词检索无法理解语义。本项目实现 RAG（检索增强生成）系统：将文档向量化存储，用户提问时语义检索相关段落，交由大模型生成准确回答，解决"AI 不知道企业内部知识"的问题。
技术架构
  • 分层架构：API 层（FastAPI 路由）→ Service 层（7 个核心服务）→ Data 层（SQLite 元数据 + ChromaDB 向量存储），层间通过 Pydantic Schema DTO 解耦
  • 数据模型：5 张 SQLAlchemy ORM 表（KnowledgeBase / Document / DocumentChunk / ChatSession / ChatMessage），支持级联删除和关联查询
  • 异步架构：FastAPI + SQLAlchemy async + aiosqlite，文档处理通过 BackgroundTasks 异步执行，不阻塞 API 响应
RAG 核心链路实现
  • 文档解析：适配 pypdf / python-docx / beautifulsoup4 / markdownify 四种解析器，统一输出纯文本
  • 智能分块：基于 LangChain RecursiveCharacterTextSplitter 实现递归字符分割，支持 chunk_size 和 chunk_overlap 参数调优，避免语义断裂
  • 向量化：Embedding 服务支持双模式——远程 OpenAI 兼容 API 或本地 ONNX 模型（all-MiniLM-L6-v2），通过 base_url 自动检测切换，降级无感知
  • 向量检索：ChromaDB 持久化存储，每个知识库独立 Collection 实现向量隔离，Cosine 相似度 Top-K 检索
  • LLM 生成：System Prompt 注入检索上下文 + RAG 指令约束（只基于资料回答、标注引用来源），temperature=0.3 降低幻觉
  • 流式输出：SSE 协议实现 token 级流式传输，先推送检索来源再流式生成回答，支持多轮对话历史（最近 6 条）
  • 引用溯源：回答附带检索命中的文档名、内容摘要和相似度分数，前端可展开查看原文
工程化实践
  • 运用 Singleton 模式管理 Embedding / VectorStore / LLM 全局实例，避免重复初始化
  • 运用 Pipeline 模式编排文档处理流水线（parse → chunk → embed → store），每步有日志和状态回写
  • 运用 Strategy 模式实现多模型适配，切换供应商只需改 .env 配置
  • Docker 多阶段构建，docker-compose 一键启动，含健康检查和自动重连
  • FastAPI 自动生成 Swagger API 文档，README 含架构图、设计模式说明和完整使用指南

Overview
Built a Retrieval-Augmented Generation (RAG) system that ingests multi-format documents, chunks and embeds them into a vector database, and answers user questions with LLM-generated responses grounded in retrieved context — with citation tracing and real-time streaming.
Key Contributions
  • Architected a 3-layer system (API → Service → Data) with 7 core service modules, 5 ORM models, and 3 RESTful route groups, decoupled via Pydantic schemas
  • Implemented the full RAG pipeline: document parsing (PDF/DOCX/TXT/MD/HTML) → recursive character splitting with overlap → embedding → ChromaDB cosine similarity Top-K retrieval → LLM streaming generation → citation tracing
  • Built SSE streaming for real-time token-by-token LLM response rendering, with multi-turn conversation context and source citation display
  • Designed a multi-provider adapter via OpenAI-compatible API, supporting DeepSeek / OpenAI / Moonshot / SiliconFlow / Ollama with zero code changes
  • Applied Singleton, Pipeline, Strategy, and DTO design patterns; implemented automatic local-to-remote embedding fallback strategy
  • Containerized with Docker Compose, including health checks and auto-reconnect; documented with Swagger API docs and comprehensive README
Results
  • Delivered 2,300+ lines across 26 files covering backend, frontend, and deployment
  • Supports 5 document formats and 5 LLM providers out of the box
  • End-to-end document ingestion to Q&A in < 10 seconds (10-page PDF)
面试官问：介绍一下这个项目
"miniKB 是我独立开发的一个 RAG 知识库问答系统，核心解决的问题是：让 AI 能基于企业内部文档准确回答问题，而不是凭空编造。

系统分三层：API 层用 FastAPI，Service 层有 7 个核心模块负责文档解析、分块、向量化、检索和生成，Data 层用 SQLite 存元数据 + ChromaDB 存向量。

用户上传文档后，系统自动解析、递归分块、Embedding 向量化存入 ChromaDB。提问时，先将问题向量化，在 ChromaDB 做 Cosine 相似度检索取 Top-K，然后把检索到的上下文拼进 System Prompt，交给 LLM 流式生成回答，最后通过 SSE 推到前端实时显示。回答会附带引用来源，可以溯源到具体文档和段落。"

→ 突出：RAG 全链路理解、架构设计能力、独立交付
面试官问：为什么用 ChromaDB 而不是 pgvector / Milvus / Pinecone？
"选 ChromaDB 主要三个原因：一是轻量，纯 Python 实现，pip install 就能用，不依赖额外服务；二是开发效率高，自带 Embedding Function 和持久化，原型阶段能快速跑通；三是每个知识库用独立 Collection 做向量隔离，删除知识库时直接删 Collection，不用维护复杂的过滤逻辑。

如果是生产环境、文档量到百万级，我会考虑切到 Milvus 做分布式向量检索，或者用 pgvector 复用现有 PostgreSQL。Pinecone 是托管服务，适合不想运维的场景，但数据要出域。"

→ 突出：技术选型有理有据、知道各方案优缺点、有生产环境意识
面试官问：怎么解决 LLM 幻觉问题？
"主要四层策略：
1. System Prompt 约束——明确要求"只根据提供的参考资料回答，不要编造信息"，如果资料中没有相关内容必须告知用户
2. 低温度生成——temperature 设为 0.3，降低随机性，让模型更忠于检索到的上下文
3. 引用溯源——回答末尾标注引用来源编号 [1] [2]，前端展示检索命中的文档名和相似度分数，用户可以验证
4. 检索质量保障——chunk_size 和 overlap 可调，Top-K 取 5，如果检索分数都很低说明知识库没有覆盖这个问题，前端会提示"

→ 突出：不只靠 prompt，有系统性的幻觉防控策略
面试官问：用了哪些设计模式？为什么要用？
"四个核心模式：
1. Singleton——embedding_service、vector_store、llm_provider 三个全局单例，避免每次请求重复创建 OpenAI client 和 ChromaDB 连接
2. Pipeline——document_pipeline 模块编排文档处理流水线（解析→分块→向量化→存储），每步有独立日志和状态回写，出错可以定位到具体步骤
3. Strategy——LLM 和 Embedding 通过 OpenAI 兼容接口实现多供应商策略，切换 DeepSeek/OpenAI/Ollama 只需改 .env 配置，不改代码
4. DTO——Pydantic Schema 和 SQLAlchemy ORM Model 分离，API 层不直接暴露数据库结构，字段可以独立演进"

→ 突出：设计模式不是背的，是解决实际问题用的
面试官问：如果文档量很大（百万级），架构怎么演进？
"几个方向：
1. 向量库升级——ChromaDB 换成 Milvus 集群，支持分布式检索和十亿级向量
2. 异步处理——文档解析和向量化走 Celery + Redis 消息队列，避免大文件阻塞 API
3. 分层检索——先做粗排（BM25 关键词召回），再做精排（向量相似度），减少向量检索的计算量
4. 缓存层——高频问题的 Embedding 向量和检索结果缓存到 Redis，避免重复计算
5. 分库分表——按知识库 ID 做水平分片，每个知识库的向量存在独立的 Collection/Partition"

→ 突出：有架构演进思维、知道当前方案的瓶颈在哪
面试官问：SSE 和 WebSocket 有什么区别？为什么选 SSE？
"SSE 是单向的（服务器→客户端），基于 HTTP，自动重连，适合 LLM 流式输出这种场景——只需要服务端推 token，客户端不需要往回发数据。

WebSocket 是双向的，适合实时聊天、协作编辑这种需要客户端也频繁推数据的场景。但 WebSocket 需要单独的连接管理、心跳维护，复杂度更高。

我这个场景：用户发一次问题，服务端流式返回回答，是典型的单向推送，SSE 足够了，而且 FastAPI 原生支持 StreamingResponse，实现很简洁。"

→ 突出：技术选型基于场景需求，不是为了用而用
